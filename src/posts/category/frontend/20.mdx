# [StoryG 실전] thumbnailUrl → imageUrls 마이그레이션

처음엔 썸네일 하나만 있으면 충분해 보였습니다. 하지만 실제 운영에서는 콘텐츠/과정/에셋에 다중 이미지 요구가 계속 들어왔고, 결국 URL 스키마를 전면 개편했습니다.

## 🚨 문제

- 단일 `thumbnailUrl`로는 다양한 화면 요구를 담기 어려움
- 에셋은 1:N 파일 구조가 필요하지만 `fileUrl` 하나만 지원
- 프론트/백엔드 응답 필드명이 화면마다 달라 유지보수 부담 증가

## 🧪 재현

```text
요구사항: 한 콘텐츠에 대표 이미지 + 보조 이미지 + 문서 첨부를 함께 관리
기존 구조: thumbnailUrl / fileUrl 단일 필드
```

## 🔍 원인 분석

문제는 기능 부족이 아니라 "데이터 모델의 확장성 한계"였습니다.

- 단일 값 중심 모델
- 리스트 렌더링 요구와 충돌
- 공통 업로드 플로우와 불일치

## 🛠️ 해결

필드를 리스트 중심으로 통일했습니다.

- `thumbnailUrl` → `imageUrls: List<String>`
- `fileUrl` → `fileUrls: List<String>`

업로드 2단계 흐름도 표준화:

```text
1) POST /files/upload or /files/upload/multiple
2) 응답 accessUrl을 생성/수정 API에 전달
```

### 예시: 콘텐츠 생성

```json
{
  "title": "SW 교육 콘텐츠",
  "imageUrls": [
    "/api/v1/files/thumbnail/a1.jpg",
    "/api/v1/files/thumbnail/b2.jpg"
  ]
}
```

### 예시: 에셋 생성

```json
{
  "contentId": 10,
  "fileUrls": [
    "/api/v1/files/document/c3.pdf",
    "/api/v1/files/image/d4.jpg"
  ],
  "fileType": "DOCUMENT",
  "title": "학습자료"
}
```

## ✅ 검증

- 콘텐츠/과정/에셋에서 다중 파일 요구 수용
- 프론트 렌더링 로직 일관성 향상
- 업로드 API와 도메인 API 계약 통일

## 📌 교훈

1. 파일 도메인은 초기에 "배열"을 기본 모델로 잡는 편이 안전합니다.
2. 필드명 통일은 기능 개발보다 장기 유지보수에 더 큰 효과가 있습니다.
3. 마이그레이션은 스키마 변경보다 클라이언트 계약 변경 관리가 핵심입니다.
