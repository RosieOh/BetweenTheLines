# [StoryG 실전] 강사 초대링크 가입 시스템

운영에서 가장 자주 들었던 요청은 "강사분들 가입 절차를 더 줄일 수 없나요?"였습니다. StoryG에서는 이 문제를 링크 기반 참여로 풀었습니다. 링크 접속 → 가입/로그인 → 과정 배정까지 한 번에 끝나는 구조입니다.

## 🚨 문제

- 강사 온보딩 시 이메일 인증/비밀번호 재설정 문의가 반복
- 과정 참여 전 단계가 많아 이탈 발생
- 운영자는 링크 발급/관리, 강사는 가입/참여를 각각 따로 처리해야 했음

## 🧪 재현

기존 흐름은 보통 이렇게 길었습니다.

1. 계정 생성
2. 로그인
3. 과정 탐색
4. 과정 참여

실제 현장에서는 2~3단계에서 자주 멈췄습니다.

## 🔍 원인 분석

핵심은 "업무 흐름과 인증 흐름이 분리"되어 있었다는 점입니다.

- 운영 관점: 특정 과정에 강사를 빠르게 배정하고 싶음
- 시스템 관점: 일반 회원가입 플로우를 먼저 요구함

현업 니즈와 시스템 절차가 충돌한 구조였습니다.

## 🛠️ 해결

링크 중심 플로우로 재설계했습니다.

```text
관리자: 초대 링크 생성
강사: 링크 접속 + 정보 입력
서버: 계정 생성(or 재사용) + 과정 배정 + JWT 쿠키 발급
```

### 초대 링크 생성 API

```http
POST /api/v1/courses/{courseId}/invite-links
```

```json
{
  "schoolName": "서울초등학교",
  "expiresAt": "2026-03-26T23:59:59"
}
```

### 링크 기반 가입 API

```http
POST /api/v1/join/{slug}/signup
```

```json
{
  "name": "홍길동",
  "phoneNumber": "010-1234-5678"
}
```

## ✅ 검증

- 신규 강사 가입 시간 단축
- 운영자 문의(아이디/비밀번호/참여 경로) 감소
- 링크별 `useCount`, 만료/삭제 상태를 운영에서 추적 가능

## 📌 교훈

1. 인증 UX는 보안만의 문제가 아니라 제품 온보딩 문제입니다.
2. B2B/교육 서비스는 "현장 플로우 우선 설계"가 중요합니다.
3. 링크 시스템은 만료/삭제/재사용 규칙까지 같이 설계해야 운영이 안정적입니다.

## ♻️ 재사용 체크리스트

- [ ] 링크 만료 정책(기본 + 최대 기간) 명시
- [ ] 소프트 삭제 지원 여부 결정
- [ ] 기존 계정 재사용 규칙(find-or-create) 정의
- [ ] 링크별 사용량/접속 로그 관측성 확보

## ✨ 마무리

이번 구조 변경에서 가장 큰 변화는 "가입 시스템을 단순화했다"가 아니라, "업무 흐름을 시스템이 따라가게 만들었다"는 점이었습니다.  
온보딩에서 자주 막히는 서비스라면, 가입 절차가 아니라 참여 시작 지점을 먼저 설계해보는 걸 추천합니다.
