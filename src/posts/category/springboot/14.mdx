# [Almighty 실전] Trap Bank 아키텍처 설계기

함정(Trap) 시나리오는 본질적으로 "정책"입니다. 그런데 초기 설계에서 Trap을 Task에 강하게 묶어두면 재사용성과 버전 관리가 어렵습니다. Almighty는 이 문제를 겪고 구조를 분리했습니다.

처음에는 과제마다 함정을 붙이는 게 가장 빠른 방법처럼 보였습니다. 하지만 운영이 길어질수록 "같은 함정을 어디에 어떻게 적용했는지" 추적하는 비용이 눈덩이처럼 커졌습니다.

## 🚨 문제

- Trap 시나리오가 Task 테이블에 종속
- 동일 Trap을 여러 Task에서 재활용하기 어려움
- 시나리오 개편 시 과거 결과와 비교가 어려움

## 🧪 재현

1. 기존 Trap을 다른 과제에서 재사용하려고 함
2. Task 종속 구조라 복제/수정 반복
3. 운영 데이터 분석 시 원본 시나리오 추적 어려움

## 🔍 원인 분석

Trap을 "콘텐츠"가 아니라 "정책 자산"으로 보지 못한 게 핵심 원인이었습니다.

- 정책 자산은 독립 저장소가 필요
- 적용 결과는 별도 이력 테이블이 필요

## 🛠️ 해결

구조를 다음처럼 재설계했습니다.

- `TBL_TRAP_BANK`: 정책 시나리오 원본 저장소
- `TBL_APPLIED_TRAP`: Task/시점별 적용 이력
- `TBL_TRAP_VERSION`: 버전 관리

이후에는 난이도/카테고리별 샘플링도 자연스럽게 구현할 수 있었습니다.

## 🔧 실전 코드: Trap 샘플링 로직

```java
public List<TrapBank> sampleTraps(Category category, Difficulty difficulty, int size) {
    List<TrapBank> candidates = trapBankRepository.findActiveByCategoryAndDifficulty(category, difficulty);
    Collections.shuffle(candidates);
    return candidates.stream().limit(size).toList();
}

public void applyTraps(Long taskId, List<TrapBank> traps) {
    for (TrapBank trap : traps) {
        appliedTrapRepository.save(
            AppliedTrap.create(taskId, trap.getTrapBankId(), "NOT_DETECTED")
        );
    }
}
```

## 🔧 실전 SQL: 정책/이력 분리 조회

```sql
SELECT a.TASK_ID, b.TRAP_TYPE, b.DIFFICULTY, a.STATUS
FROM TBL_APPLIED_TRAP a
JOIN TBL_TRAP_BANK b ON a.TRAP_BANK_ID = b.TRAP_BANK_ID
WHERE a.TASK_ID = :taskId;
```

## ✅ 검증

- Trap 재사용률 증가
- 시나리오 변경 이력 추적 가능
- 과거/현재 버전 비교 리포트 작성 가능

## 📌 교훈

1. 정책 데이터와 실행 이력은 같은 테이블에 두지 않는 것이 좋습니다.
2. 재사용성과 추적 가능성은 분리 설계에서 나옵니다.
3. 버전 없는 보안/평가 정책은 운영 리스크를 키웁니다.

## ♻️ 재사용 프레임

- 정책 테이블(원본) + 적용 테이블(실행) + 버전 테이블(이력)
- 정책 변경은 append 방식으로 관리
- 실험/운영 환경 분리 가능하도록 설계

## ✨ 마무리

정책성 데이터는 반드시 장기 관점으로 설계해야 합니다. 당장의 구현 속도만 보고 결합도를 높이면, 결국 분석/운영 단계에서 큰 비용을 치르게 됩니다.  
Almighty에서는 Trap 분리 이후 실험 설계와 결과 해석이 훨씬 명확해졌습니다.
