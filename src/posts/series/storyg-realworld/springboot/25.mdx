# [StoryG 프로젝트] AI 리포트 프롬프트 운영 패턴 7가지

`aiResult`를 넣은 뒤 분명 품질은 좋아졌습니다. 그런데 운영자는 또 다른 피드백을 주었습니다.  
"어제는 좋았는데 오늘은 톤이 좀 달라요." 모델 품질이 아니라 운영 일관성의 문제였습니다.

여기서 배운 점은 명확했습니다. 프롬프트는 글쓰기 기술이 아니라 운영 규칙이라는 사실입니다.  
입력 우선순위, 출력 포맷, 후처리 책임이 문서화되지 않으면 사람마다 기준이 달라지고, 결국 제품 품질이 흔들립니다.

이 글은 그때 만든 "팀 공통 규칙 7가지"를 실무 관점으로 정리한 운영 가이드입니다.

## 🕰️ 사건 타임라인

- 1차 개선: `aiResult` 적용으로 품질 평균치 상승
- 2차 이슈: 출력 스타일 편차/후처리 누락 반복
- 대응: 프롬프트 운영 규칙 7개를 문서/코드로 고정
- 결과: 출력 구조/인용 규칙 일관성 확보

## 🚨 문제

- 같은 조건인데 결과 문체/깊이가 들쭉날쭉
- 리포트에 학생 원문 인용 비율이 일정하지 않음
- 마크다운 표기(`**`) 같은 후처리 이슈가 반복

## 🧪 재현

- 동일 과정/유사 응답 입력에서도 출력 길이 편차 큼
- 일부 출력은 교육 맥락보다 일반 표현 위주

## 🔍 원인 분석

원인은 모델 자체보다 프롬프트 운영 규칙 부족이었습니다.

- 입력 필드 우선순위 불명확
- 출력 포맷 제약 약함
- 후처리 계약(인용 표기, 마크다운 제거) 미고정

## 🛠️ 해결

아래 7가지 패턴을 고정 규칙으로 도입했습니다.

1. **컨텍스트 계층화**: 과정(`aiResult`) → 학생 응답 → 출력 지시 순
2. **출력 섹션 고정**: 요약/강점/성장/지도포인트 순서 강제
3. **인용 규칙 고정**: 학생 원문은 `"..."(학생원문)` 형태
4. **금지 토큰 명시**: `**`, `*` 마크다운 금지
5. **길이 가이드**: 섹션별 최소/최대 문장 수 지정
6. **fallback 규칙**: `aiResult` 없으면 일반 안내문 삽입
7. **후처리 단일화**: 서버에서 일괄 정제, 프론트 무처리

예시 프롬프트 뼈대:

```text
[과정 참고 정보]
{aiResult or defaultContext}

[출력 규칙]
1) 섹션 순서: 요약-강점-성장-지도포인트
2) 학생 원문 인용은 "..."(학생원문) 형식만 허용
3) markdown 기호(**, *) 사용 금지
```

## ✅ 검증

- 섹션 구조 일관성 향상
- 인용 표기 누락 감소
- 프론트 후처리 코드 제거 가능

## 📌 교훈

1. 좋은 프롬프트는 "잘 쓰는 문장"보다 "운영 규칙"에서 나옵니다.
2. 출력 포맷이 제품 계약이라면, 프롬프트도 API처럼 버전 관리해야 합니다.
3. 후처리 책임을 분산하면 장애 분석이 어려워집니다.
