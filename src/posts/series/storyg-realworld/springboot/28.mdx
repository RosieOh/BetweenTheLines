# [StoryG 프로젝트] DB Deadlock/Lock Timeout 실전 해결기

배치 보고서 생성은 성공했다가도 특정 시점부터 실패가 반복됐습니다. 로그에는 `1213`, `1205`가 번갈아 찍혔고, 운영에서는 재시도 요청이 쌓이면서 큐 지연이 커졌습니다.  
이번 글은 StoryG에서 deadlock/lock timeout을 "SQL 한 줄 튜닝"이 아닌 **트랜잭션 구조 개선**으로 해결한 과정을 정리합니다.

처음에는 DB 인덱스 문제라고 생각했습니다. 실제로 인덱스도 점검했고, 일부 쿼리도 손봤습니다.  
그런데 증상은 잠시 줄었다가 다시 올라왔습니다. 결국 "쿼리 튜닝으로 가릴 수 없는 구조 문제"라는 결론에 도달했습니다.

특히 운영에서 어려웠던 점은, 실패가 항상 같은 지점에서 일어나지 않는다는 점이었습니다.  
어떤 날은 deadlock, 어떤 날은 timeout, 또 어떤 날은 재시도 후 성공. 이 변동성 때문에 원인 추적이 오래 걸렸습니다.

## 🕰️ 사건 타임라인

- 배치 재생성 트래픽 증가와 함께 1213/1205 동시 관측
- 1차 대응: 인덱스/쿼리 미세조정(효과 제한적)
- 2차 대응: 트랜잭션 경계/동시성 모델 재설계
- 결과: 수동 재실행 중심 운영에서 자동 복구 중심 운영으로 전환

## 🚨 문제

- 배치 보고서 생성 중 `Deadlock found when trying to get lock` 발생
- 이후 `Lock wait timeout exceeded`로 연쇄 실패
- 동일 `programId` 재생성 요청에서 실패율 급증

사용자에게는 "버튼을 눌렀는데 완료되지 않음", 운영에는 "수동 재실행 증가"로 나타났습니다.

## 🧪 재현

다음 조건이 겹치면 높은 확률로 재현되었습니다.

1. 동일 `programId` 대상 배치를 병렬 스레드로 실행
2. `@Transactional` 내부에서 `DELETE` 수행 후 외부 AI 호출
3. 같은 키 범위에 대해 `INSERT/UPDATE`가 뒤이어 경합

대표 로그:

```text
Deadlock found when trying to get lock; try restarting transaction
Lock wait timeout exceeded; try restarting transaction
```

## 🔍 원인 분석

문제는 쿼리 하나가 아니라 실행 모델이었습니다.

### 1) Deadlock (1213)

- 트랜잭션 A/B가 동일 리소스를 다른 순서로 잠금
- 상호 대기 상태에 진입
- InnoDB가 한쪽 트랜잭션을 강제 중단

### 2) Lock wait timeout (1205)

- `DELETE`가 획득한 락이 커밋까지 유지
- 같은 트랜잭션 내부에서 AI 호출로 지연 발생
- 대기 중 트랜잭션이 timeout 도달

### 3) 왜 반복됐나

- `DELETE -> 외부 호출 -> 재삽입` 패턴이 여러 경로에서 재사용
- 동일 키 동시 실행을 제한하는 장치 부재

즉, deadlock/timeout은 "일시적 예외"가 아니라 "구조적 충돌 신호"였습니다.

## 🛠️ 해결

핵심은 락 점유 시간과 동시 경합 자체를 줄이는 것이었습니다.

### 1) 트랜잭션 경계 축소

- 외부 AI 호출을 트랜잭션 밖으로 이동
- DB 쓰기 구간만 짧은 트랜잭션으로 분리

효과: 락 점유 시간 감소.

### 2) 동일 키 직렬화

- 동일 `programId` 작업은 동시에 실행하지 않도록 제어
- 배치 큐에서 키 단위 실행 순서 보장

효과: 교착 발생 가능성이 높은 경합 구간 제거.

### 3) 삭제 중심 흐름 완화

- 대량 `DELETE` 후 재삽입 패턴 최소화
- 가능 구간은 상태 전환/업서트 방식으로 대체

효과: gap lock 빈도 감소.

### 4) 재시도 정책 표준화

- deadlock/timeout을 재시도 대상 예외로 분류
- 짧은 backoff + 제한된 최대 재시도 횟수 적용

효과: 일시 충돌의 자동 복구율 상승.

## ✅ 검증

운영에서는 아래 지표로 효과를 검증했습니다.

- `1213/1205` 오류 발생 빈도
- 배치 평균 완료 시간과 p95
- 동일 `programId` 재생성 최종 성공률
- 수동 재실행 건수

정량 수치는 환경마다 다르지만, "반복 실패 -> 자동 복구" 흐름으로 전환되며 운영 부담이 확실히 줄었습니다.

## 📌 교훈

1. 외부 API/AI 호출은 트랜잭션 안에 두지 않는다.
2. 동일 비즈니스 키 쓰기 작업은 직렬화 가능성을 먼저 검토한다.
3. deadlock/timeout은 예외 처리보다 구조 개선 신호로 본다.
