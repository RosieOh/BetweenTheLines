# [StoryG 프로젝트] 2270줄 God Class 분해기

운영 이슈가 반복될수록 같은 결론에 도달했습니다. "버그를 고친다"보다 먼저 "고칠 수 있는 구조"를 만들어야 한다는 점입니다.  
StoryG에서는 `ReportService`가 2270줄까지 커지며 변경 영향 범위를 예측하기 어려운 상태였고, 테스트도 거의 없는 상황이었습니다.

개발자 입장에서 가장 힘들었던 건 코드 길이 자체가 아니었습니다.  
한 줄을 바꾸면 어떤 기능이 깨질지 누구도 확신할 수 없다는 불안이 더 큰 문제였습니다.

그래서 이번 리팩토링은 "예쁘게 정리"가 아니라 "운영 리스크를 줄이는 구조 변경"으로 접근했습니다.  
한 번에 갈아엎지 않고, 장애 가능성이 높은 경로부터 작은 단위로 분해해 실제 서비스에 안전하게 적용했습니다.

## 🕰️ 사건 타임라인

- 반복 회귀 이슈에서 ReportService 집중도 확인
- 책임 분해 설계(Query/Generation/Persistence/AI) 수립
- 작은 PR 단위로 단계적 분리 + 기능 동등성 검증
- 핵심 경로 테스트 도입으로 배포 전 불확실성 축소

## 🚨 문제

- 단일 서비스 클래스에 생성/조회/가공/AI 호출/저장 로직이 과도하게 집중
- 작은 수정도 사이드이펙트 위험이 큼
- 테스트 코드 부재로 배포 전 회귀 확인이 어려움

결과적으로 개발 속도보다 "변경 불안"이 더 큰 비용이 되었습니다.

## 🧪 재현

다음 패턴이 반복됐습니다.

1. 특정 기능 수정
2. 예상 못한 다른 리포트 유형에서 회귀 발생
3. 원인 추적 시 하나의 거대 클래스 내부 분기/조건문으로 확산

즉, 장애 원인은 단일 버그가 아니라 설계 복잡도 누적이었습니다.

## 🔍 원인 분석

God Class가 된 이유는 명확했습니다.

- 도메인 경계 없이 기능을 계속 추가
- 공통 로직과 유형별 로직이 같은 계층에 혼재
- 외부 의존(AI/DB/매퍼) 호출 책임이 분리되지 않음

테스트가 없는 이유도 동일했습니다.  
의존이 너무 많아 단위 테스트를 쓰기 어려운 구조였습니다.

## 🛠️ 해결

핵심 목표는 "한 번에 완전 교체"가 아니라 **점진적 분해 + 안전한 검증**이었습니다.

### 1) 책임 단위로 서비스 분리

- `ReportQueryService`: 조회/필터링
- `ReportGenerationService`: 생성 규칙/조합
- `ReportPersistenceService`: 저장/갱신
- `ReportAiService`: AI 호출/후처리

기존 `ReportService`는 orchestration 역할만 남기고 점진 축소했습니다.

### 2) 공통 규칙 추출

- 중복된 변환/검증 코드를 유틸 또는 도메인 정책 클래스로 이동
- 리포트 유형별 분기문은 전략 패턴 형태로 분리

### 3) 테스트 우선순위 설정

처음부터 100% 커버리지를 목표로 하지 않고, 장애 재발 가능성이 큰 경로부터 테스트를 추가했습니다.

1. 생성 핵심 시나리오(정상 경로)
2. 경계 조건(빈 데이터, null, 중복 요청)
3. 회귀 이력 이슈(deadlock/stale 관련 후속 흐름)

### 4) 배포 안정화 전략

- 큰 리팩토링 대신 작은 PR 단위로 분할
- 기능 동등성 확인 체크리스트 운영
- 운영 지표(실패율/생성시간) 모니터링으로 즉시 회귀 감지

## ✅ 검증

적용 후 확인된 변화:

- 변경 범위 예측 가능성 개선
- 장애 원인 추적 시간 단축
- 신규 기능 추가 시 영향도 감소
- 테스트 기반으로 배포 전 검증 품질 향상

수치보다 중요한 변화는, 팀이 "수정이 두렵지 않은 코드"로 전환되기 시작했다는 점이었습니다.

## 📌 교훈

1. God Class는 코드 냄새가 아니라 운영 리스크다.
2. 리팩토링은 구조 개선과 테스트 도입이 함께 가야 효과가 난다.
3. 대규모 교체보다 점진 분해가 실제 서비스에 더 안전하다.

## ♻️ 재사용 체크리스트

- [ ] 단일 클래스의 책임이 3개 이상인지 점검
- [ ] 외부 의존 호출 책임(AI/DB/API) 분리
- [ ] 핵심 시나리오 테스트 먼저 작성
- [ ] 리팩토링 PR 크기를 작게 유지
- [ ] 배포 후 실패율/처리시간 회귀 모니터링
