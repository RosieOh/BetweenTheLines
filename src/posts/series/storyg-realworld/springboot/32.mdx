# [StoryG 프로젝트] N+1과 페이지네이션 개선기

초기에는 조회 API가 "기능상 정상"이었지만, 데이터가 늘자 응답 시간이 급격히 느려졌습니다.  
분석해보니 핵심 원인은 `N+1 쿼리`와 `무제한 조회`가 겹친 구조였습니다.

운영에서는 이런 이슈가 가장 무섭습니다. 장애처럼 완전히 멈추지 않아서, 우선순위에서 계속 밀리기 때문입니다.  
"가끔 느리다"는 제보가 쌓일 때쯤이면 이미 사용자는 제품을 느린 서비스로 인식하기 시작합니다.

이번 개선의 목표는 체감 성능을 올리는 것뿐 아니라, 데이터가 더 늘어도 성능이 무너지지 않는 구조를 만드는 것이었습니다.

## 🕰️ 사건 타임라인

- 사용자 제보: 특정 시간대 목록 조회 지연 증가
- 관측: 쿼리 수 폭증 + 전체 조회 응답 비대화
- 개선: DTO 기반 조회/페이지네이션 표준화 적용
- 검증: 쿼리 수, p95, 슬로우쿼리 지표 기반 운영 확인

## 🚨 문제

- 목록 조회 시 관련 엔티티를 건수만큼 추가 조회하는 N+1 발생
- 전체 조회 API에 페이지네이션 미적용
- 데이터가 늘수록 응답 지연과 DB 부하가 같이 증가

운영에서 체감된 증상은 "가끔 느리다"가 아니라, 특정 시간대엔 "목록 페이지가 버벅인다"였습니다.

## 🧪 재현

다음 상황에서 병목이 분명하게 드러났습니다.

1. 게시물/리포트 목록을 다건 조회
2. 각 항목에서 연관 데이터(작성자, 태그, 통계 등)를 지연 로딩
3. API가 페이지 없이 전체 결과를 반환

대표 패턴:

```text
1번 목록 조회 + N번 연관 조회 = 총 N+1 쿼리
```

조회 건수가 늘어날수록 쿼리 수가 선형 이상으로 증가했습니다.

## 🔍 원인 분석

### 1) N+1 쿼리

- 루트 엔티티 조회 후 연관 엔티티를 건별 추가 조회
- ORM 기본 동작에 의존하면서 쿼리 폭증

### 2) 페이지네이션 부재

- API가 "필요한 만큼"이 아닌 "있는 만큼 전부" 반환
- 네트워크/직렬화/DB 처리 비용이 한 번에 집중

결국 병목은 쿼리 자체보다 "조회 설계" 문제였습니다.

## 🛠️ 해결

### 1) 조회 쿼리 재설계

- 필요한 화면 데이터 기준으로 조회용 쿼리 분리
- `fetch join` 또는 배치 조회 전략으로 N+1 제거
- 엔티티 전체 대신 DTO 프로젝션을 우선 적용

예시(개념):

```java
@Query("""
select new com.storyg.api.dto.PostListItemDto(
  p.id, p.title, a.name, p.createdAt
)
from Post p
join p.author a
where p.status = 'PUBLISHED'
order by p.createdAt desc
""")
Page<PostListItemDto> findPublishedPostItems(Pageable pageable);
```

### 2) 페이지네이션 표준화

- 기본 `page`, `size`, `sort` 파라미터 도입
- 최대 page size 제한으로 과도한 요청 차단
- total count/next page 정보 포함

### 3) 인덱스 및 정렬 기준 점검

- 자주 쓰는 필터/정렬 컬럼 인덱스 점검
- 정렬/검색 조건과 실제 인덱스 사용 여부 확인

### 4) 성능 회귀 방지

- 주요 조회 API 쿼리 수/응답 시간 모니터링
- 배포 후 샘플 데이터셋 기준 성능 체크 자동화

## ✅ 검증

개선 효과는 아래 지표로 확인했습니다.

- 요청 1건당 실행 쿼리 수
- 평균 응답 시간 및 p95
- DB CPU/슬로우 쿼리 비율
- 동시 접속 시 처리량

정량 수치는 환경마다 다르지만, 운영에서는 "데이터가 늘면 느려진다" 패턴이 크게 완화되었습니다.

## 📌 교훈

1. N+1은 코드 리뷰에서 놓치기 쉬워도 운영에서 반드시 비용으로 돌아온다.
2. 페이지네이션은 옵션이 아니라 조회 API의 기본 계약이다.
3. "빠른 조회"는 쿼리 튜닝보다 반환 모델 설계에서 시작된다.
