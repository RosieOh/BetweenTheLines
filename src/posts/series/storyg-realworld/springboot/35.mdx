# [StoryG 프로젝트] AI 재생성 불능 이슈 해결기

운영에서 가장 답답한 버그 중 하나는 "재생성 버튼은 눌리는데 결과가 바뀌지 않는 문제"였습니다.  
로그는 정상처럼 보였지만, 실제로는 stale 캐시/트랜잭션 경계 문제로 이전 상태를 계속 반환하고 있었습니다.

이 이슈가 특히 어려웠던 이유는, 실패 로그가 명확하지 않았기 때문입니다.  
"완료" 로그는 찍히는데 화면 결과는 그대로였고, 운영자는 같은 버튼을 여러 번 누르게 됐습니다.

초기에는 AI 모델 품질 문제로 오해하기 쉬웠지만, 실제 원인은 상태 전이 설계였습니다.  
재생성 요청이 들어왔을 때 캐시/트랜잭션/저장/재조회가 어떤 순서로 보장되는지 계약이 없었던 겁니다.

## 🕰️ 사건 타임라인

- 운영 제보: 재생성 성공 로그 대비 결과 미변경
- 원인 추적: stale 캐시 hit + 프록시 우회 호출 확인
- 개선: invalidate/regenerate/persist/reload 단계 분리
- 검증: 재생성 후 최신값 보장 회귀 테스트로 안정화

## 🚨 문제

- AI 리포트를 재생성해도 이전 결과가 그대로 보임
- 일부 케이스에서 AI 재호출 자체가 일어나지 않음
- 운영자는 재시도를 반복하지만 결과는 동일

사용자 관점에서는 "기능이 동작하지 않는다", 운영 관점에서는 "문제가 재현되지만 원인이 안 보인다" 상태였습니다.

## 🧪 재현

다음 흐름에서 증상이 반복됐습니다.

1. 기존 결과 삭제 또는 재생성 요청
2. 내부 호출에서 self-invocation/전파 조합으로 기존 트랜잭션 문맥 합류
3. 캐시 또는 stale 스냅샷을 다시 참조

대표 증상:

```text
재생성 완료 로그는 존재
하지만 조회 결과는 이전 값과 동일
```

## 🔍 원인 분석

핵심 원인은 두 축이 결합된 구조였습니다.

### 1) 캐시 stale

- 재생성 이전 데이터가 캐시에서 무효화되지 않음
- 삭제/재생성 후 조회가 최신 DB 대신 stale 캐시를 반환

### 2) 호출/트랜잭션 경계 문제

- self-invocation으로 의도한 분리 트랜잭션이 적용되지 않음
- `REQUIRED` 합류로 stale 상태를 재사용
- 결과적으로 AI 재호출 조건 판단이 틀어짐

즉 "재생성 API 호출"과 "실제 최신 데이터 계산" 사이에 경계 불일치가 있었습니다.

## 🛠️ 해결

### 1) 재생성 플로우를 단계별로 분리

- invalidate(캐시 무효화)
- regenerate(AI 호출/계산)
- persist(저장)
- reload(최신값 재조회)

각 단계의 책임과 트랜잭션 경계를 명확히 분리했습니다.

### 2) 캐시 무효화 타이밍 고정

- 재생성 시작 전에 관련 키를 선제 무효화
- 저장 완료 후 최종 상태를 다시 캐시에 반영

핵심은 "삭제 후 무효화"가 아니라 "재생성 흐름의 계약으로 무효화"를 강제하는 것이었습니다.

### 3) 프록시 경유 호출 보장

- self-invocation 제거, 별도 빈 호출로 전환
- 전파 정책을 단순화해 stale 문맥 합류를 최소화

### 4) 후처리 품질 고정

- AI 응답 후 마크다운/불필요 토큰 정규화
- 동일 입력/동일 규칙일 때 일관된 저장 포맷 유지

## ✅ 검증

개선 후 아래 항목을 검증했습니다.

- 재생성 직후 조회 값의 실제 변경 여부
- 캐시 hit/miss 비율과 stale hit 감소
- AI 재호출 누락 케이스 제거 여부
- 동일 시나리오 반복 시 결과 일관성

운영에서는 "재생성했는데 그대로" 유형의 문의가 눈에 띄게 줄었습니다.

## 📌 교훈

1. 캐시는 성능 도구이지만, 무효화 계약이 없으면 장애 증폭기가 된다.
2. 재생성 기능은 API 하나가 아니라 상태 전이(state transition)로 설계해야 한다.
3. AI 품질 이슈처럼 보이는 문제도 상당수는 캐시/트랜잭션 경계 문제다.

## ♻️ 재사용 체크리스트

- [ ] 재생성 플로우의 단계별 책임 분리(invalidate/regenerate/persist/reload)
- [ ] 캐시 무효화 시점과 키 전략 문서화
- [ ] self-invocation 여부 및 프록시 경유 호출 점검
- [ ] 전파 속성 단순화(REQUIRED/REQUIRES_NEW 최소화)
- [ ] 재생성 회귀 테스트(결과 변경/최신값 조회) 보유
