# [StoryG 프로젝트] AI 프롬프트 엔지니어링

AI 리포트 품질이 일정하지 않다는 피드백을 받을 때, 처음엔 모델 문제라고 생각하기 쉽습니다. 그런데 StoryG에서 파고 들어보니 핵심은 모델보다 컨텍스트였습니다. 그래서 `aiResult` 필드를 도입해 프롬프트 입력 품질을 먼저 올렸습니다.

## 🚨 문제

- 학생 답변 데이터는 충분한데 리포트가 너무 일반론적으로 생성됨
- 과정마다 교육 목표가 다른데 출력 문장이 비슷하게 반복됨
- 운영자가 의도한 "과정 맥락"이 리포트에 반영되지 않음

## 🧪 재현

동일 학생 응답 데이터를 서로 다른 과정에 적용해도 출력이 유사한 패턴이 나왔습니다.

```text
입력: 학생 답변 중심 데이터만 전달
출력: 과정별 차이가 약한 일반적 코멘트
```

## 🔍 원인 분석

프롬프트의 도메인 컨텍스트가 비어 있었습니다.

- 있었던 것: 학생별 응답 데이터
- 없었던 것: 과정의 목적/의도/기대 역량

즉 모델은 "무엇을 답했는지"는 알지만 "왜 이 활동을 했는지"는 모르는 상태였습니다.

## 🛠️ 해결

과정 CRUD에 `aiResult`를 추가하고, 리포트 생성 프롬프트에 자동 삽입했습니다.

### 1) DB 컬럼 추가

```sql
ALTER TABLE tbl_courses ADD COLUMN ai_result LONGTEXT NULL;
```

### 2) 과정 생성/수정에서 컨텍스트 입력

```json
{
  "title": "2026 진로탐구 과정",
  "aiResult": "이 과정은 학생들이 자신의 흥미와 적성을 발견하고 진로를 구체화하는 데 초점을 맞춥니다.",
  "startDatetime": "2026-03-01T09:00:00",
  "endDatetime": "2026-06-30T18:00:00"
}
```

### 3) 프롬프트 템플릿에 주입

```text
[과정 참고 정보]
{course.aiResult}

[학생 응답]
활동 계기: ...
활동 과정: ...
성장 포인트: ...
```

`aiResult`가 없는 경우 해당 섹션은 생략해 기존 동작과 호환되도록 유지했습니다.

## ✅ 검증

- 과정별 맥락 차이가 출력 문장에 명확히 반영
- 운영자 피드백: "리포트가 과정 의도를 더 잘 설명한다"
- 모델 변경 없이도 품질 체감 개선

## 📌 교훈

1. 프롬프트 엔지니어링은 문장 기술이 아니라 데이터 설계 문제입니다.
2. 운영자가 컨텍스트를 제어할 수 있어야 AI 결과의 품질을 관리할 수 있습니다.
3. nullable 필드로 도입하면 위험 없이 점진 적용이 가능합니다.

## ♻️ 재사용 체크리스트

- [ ] 모델 변경 전, 프롬프트 입력 데이터 충분성 점검
- [ ] 도메인 컨텍스트 필드(예: aiResult) 별도 설계
- [ ] 컨텍스트 미입력 fallback 동작 정의
- [ ] 변경 전/후 출력 샘플 비교 로그 수집

## ✨ 마무리

모델을 바꾸기 전에, 먼저 컨텍스트를 더 잘 전달할 수 있는 프롬프트 입력 구조를 설계해보세요.
