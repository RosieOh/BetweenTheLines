# [StoryG 프로젝트] Snapshot 검증 SQL과 품질 게이트

Snapshot 저장 로직을 적용한 뒤, 팀 분위기는 잠깐 안도감이 있었습니다. "이제 해결됐다"는 느낌이었죠.  
그런데 리포트팀은 한 문장을 남겼습니다. "구조는 좋아졌는데, 매 배포마다 믿어도 되는 데이터인지 확인이 어렵습니다."

그 순간부터 목표가 바뀌었습니다. 저장 성공이 아니라 신뢰 가능한 운영 데이터가 목표가 됐습니다.  
기능 구현을 완료하는 단계에서, 품질 회귀를 자동으로 감지하는 단계로 넘어가야 했습니다.

이 글은 PRE/POST 데이터를 "모아두는 것"에서 "계속 신뢰할 수 있게 지키는 것"으로 전환한 과정을 담았습니다.

## 🕰️ 사건 타임라인

- Snapshot 저장 구조 도입 완료
- 배포 이후 품질 검증 수동 운영으로 병목 발생
- 핵심/누락 탐지 SQL 고정
- 품질 게이트와 배포 후 점검 루틴 정착

## 🚨 문제

- 기능은 동작하지만 데이터 품질 회귀를 빨리 감지하기 어려움
- 특정 배포 이후 `response_context` 누락 가능성 우려
- 리포트팀이 데이터 신뢰 여부를 수동으로 확인해야 하는 부담

## 🧪 재현

운영 배포 후 다음 징후가 나타나면 위험 신호입니다.

- PRE/POST 응답 수 불균형
- `isPrePost=true` 질문인데 `response_context`가 `NULL`
- 코스별 특정 구간에서만 전후 데이터 누락

## 🔍 원인 분석

Snapshot 품질은 저장 로직만으로 보장되지 않습니다.

- 템플릿 변경
- 프론트 요청 누락
- 서비스 리팩터링

이 세 가지가 언제든 품질을 깨뜨릴 수 있습니다.

## 🛠️ 해결

### 1) 핵심 검증 SQL 고정

```sql
SELECT a.question_id, a.response_context, COUNT(*) AS cnt
FROM tbl_answers a
JOIN tbl_survey_responses r ON a.survey_response_id = r.id
WHERE r.course_id = :courseId
  AND r.is_satisfaction = 1
GROUP BY a.question_id, a.response_context
ORDER BY a.question_id, a.response_context;
```

### 2) 누락 탐지 SQL

```sql
SELECT a.question_id, COUNT(*) AS null_count
FROM tbl_answers a
JOIN tbl_survey_responses r ON a.survey_response_id = r.id
WHERE r.is_satisfaction = 1
  AND a.response_context IS NULL
GROUP BY a.question_id
HAVING COUNT(*) > 0;
```

### 3) 품질 게이트 기준

- `isPrePost=true` 문항의 `NULL response_context` 비율 0% 유지
- PRE/POST 개수 편차 임계치 초과 시 알림
- 배포 후 샘플 코스 자동 점검

## ✅ 검증

- 배포 직후 품질 이상을 조기 탐지 가능
- 리포트팀/분석팀의 수동 확인 시간 감소
- "데이터가 맞다"를 숫자로 설명할 수 있게 됨

## 📌 교훈

1. 데이터 기능은 구현으로 끝나지 않고 검증 체계까지 있어야 완성됩니다.
2. 품질 게이트는 장애 대응보다 훨씬 저렴합니다.
3. SQL 검증은 운영에서 가장 빠르고 확실한 안전장치입니다.
