# [StoryG 프로젝트] @Async self-invocation 함정

StoryG 배치 리포트 로직에서 분명 `@Async`를 붙였는데, 실제로는 동기처럼 동작하는 구간이 있었습니다.  
문제는 단순 성능 저하로 끝나지 않았고, 트랜잭션 전파와 결합되며 stale 데이터/낙관적 락 충돌까지 이어졌습니다.

가장 혼란스러웠던 건 로그였습니다. 코드만 보면 비동기인데, 타임라인을 보면 순차 실행처럼 보였습니다.  
처음엔 스레드풀 설정을 의심했고, 다음엔 DB를 의심했습니다. 하지만 핵심은 그 사이가 아니라 호출 경로 자체였습니다.

이 이슈를 겪고 팀에서 가장 크게 바뀐 관점은 이것입니다.  
"어노테이션을 붙였다"가 아니라 "프록시를 타도록 설계했는가"를 먼저 확인하게 됐습니다.

## 🕰️ 사건 타임라인

- 증상 제보: 비동기 처리인데 응답 지연/갱신 불일치
- 1차 분석: 스레드풀/DB 상태 점검(직접 원인 아님)
- 2차 분석: self-invocation 프록시 우회 확인
- 개선: 호출 구조 분리 + 전파 정책 단순화 + 회귀 테스트 추가

## 🚨 문제

- `@Async` 메서드가 기대처럼 별도 스레드에서 분리 실행되지 않음
- 재생성 로직에서 stale 엔티티를 업데이트하며 충돌 발생
- 일부 케이스에서 "재생성했는데 결과가 갱신되지 않음" 현상 반복

## 🧪 재현

재현 패턴은 아래와 같았습니다.

1. 서비스 내부 메서드에서 같은 클래스의 `@Async` 메서드를 직접 호출
2. 호출 체인에 `REQUIRED`/`REQUIRES_NEW`가 혼재
3. DELETE/재생성/UPDATE 흐름이 한 트랜잭션 문맥에서 얽힘

증상:

- 비동기 기대 구간이 호출 스레드에서 즉시 실행
- 갱신 직후 재조회 시 이전 스냅샷이 보이는 케이스 발생

## 🔍 원인 분석

핵심은 Spring AOP 프록시 동작 원리였습니다.

### 1) self-invocation

- `@Async`, `@Transactional`은 프록시를 경유할 때 적용
- 같은 클래스 내부에서 `this.asyncMethod()` 호출 시 프록시 우회
- 결과적으로 비동기/전파 규칙이 의도대로 발동하지 않음

### 2) 전파 속성과 stale 데이터

- `REQUIRED`는 기존 트랜잭션에 합류
- `REQUIRES_NEW`는 새 트랜잭션 시작
- 의도와 다르게 합류/분리가 되면 조회 스냅샷과 업데이트 시점이 꼬이며 stale 상태 유발

즉, 이슈의 본질은 "어노테이션 유무"가 아니라 "프록시를 타는 호출 경로 설계"였습니다.

## 🛠️ 해결

### 1) 비동기 책임을 별도 빈으로 분리

- `AsyncFacade` 또는 전담 서비스로 분리
- 외부 빈 호출 구조로 프록시 경유 보장

```java
// bad: same class internal call
this.generateReportAsync(programId);

// good: separate bean call
reportAsyncFacade.generateReportAsync(programId);
```

### 2) 트랜잭션 경계 재정의

- 읽기/삭제/생성 단계를 명확히 분리
- 장시간 작업(AI 호출, 외부 I/O)은 트랜잭션 바깥으로 이동

### 3) 전파 규칙 단순화

- 꼭 필요한 구간에만 `REQUIRES_NEW` 사용
- 기본은 예측 가능한 `REQUIRED`로 유지

### 4) 회귀 테스트 추가

- 비동기 실행 여부(스레드/타이밍) 검증
- 재생성 후 최신값 반영 검증
- 낙관적 락 충돌 시 재시도/실패 처리 검증

## ✅ 검증

적용 후 아래 지표를 확인했습니다.

- 비동기 구간의 실제 스레드 분리 여부
- 재생성 후 데이터 최신성(stale 비율) 감소
- 낙관적 락 예외 발생 건수 감소

결과적으로 "실행은 됐지만 결과는 갱신되지 않는" 종류의 난해한 장애가 크게 줄었습니다.

## 📌 교훈

1. `@Async`는 어노테이션이 아니라 호출 경로 설계 문제다.
2. 트랜잭션 전파는 많을수록 안전한 게 아니라, 단순할수록 예측 가능하다.
3. self-invocation이 보이면 먼저 프록시 우회 여부를 의심해야 한다.

## ♻️ 재사용 체크리스트

- [ ] `@Async` 메서드가 외부 빈 호출로 진입하는지 확인
- [ ] `@Transactional` 전파 조합(REQUIRED/REQUIRES_NEW) 최소화
- [ ] 장시간 I/O가 트랜잭션 내부에 있는지 점검
- [ ] 재생성 후 최신 데이터 검증 테스트 보유
- [ ] 낙관적 락 예외 처리/재시도 정책 점검
