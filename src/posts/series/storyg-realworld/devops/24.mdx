# [StoryG 프로젝트] Nginx 트러블슈팅 모음

Nginx 이슈는 로그 한 줄로는 원인이 안 보이는 경우가 많습니다. StoryG에서도 `502 → 403 → 405`가 연달아 터지면서, 결국 프록시/도메인/context-path를 하나의 시스템으로 봐야 한다는 걸 배웠습니다.

## 🚨 문제

- `GET /api/v1/...` 호출 시 `502 Bad Gateway`
- Swagger에서 POST 테스트 시 `405 Method Not Allowed`
- 배포 후 일부 경로만 정상, 일부 경로는 정적 서버로 라우팅

## 🧪 재현

대표 실패 흐름:

```text
POST http://ourstoryg.kr/api/v1/auth/login
→ nginx 301 redirect to https
→ browser converts POST to GET
→ backend @PostMapping 미매칭
→ 405
```

## 🔍 원인 분석

원인은 한 가지가 아니었습니다.

1. `VIRTUAL_HOST` 누락으로 upstream 자동 매핑 실패 (`502`)
2. `SERVER_URL`에 context-path(`/api/v1`) 누락 (`405`)
3. prod context-path가 `/api`로 달라 프론트 요청(`/api/v1/...`)과 불일치 (`404/405`)

## 🛠️ 해결

### 1) Jenkins `docker run`에 vhost 환경변수 명시

```groovy
-e VIRTUAL_HOST=api.ourstoryg.kr \
-e VIRTUAL_PORT=8080 \
```

### 2) Swagger 서버 URL을 HTTPS + context-path까지 명시

```groovy
-e SERVER_URL=https://ourstoryg.kr/api/v1 \
```

```yaml
app:
  server-url: ${SERVER_URL:https://ourstoryg.kr/api/v1}
```

### 3) prod context-path 통일

```yaml
server:
  servlet:
    context-path: /api/v1
```

## ✅ 검증

- `GET /api/v1/courses/public` 정상 응답
- Swagger POST 호출 정상 처리
- 프론트/백엔드/프록시 라우팅 경로 일치

## 📌 교훈

1. Nginx 장애는 프록시 설정만의 문제가 아니라 배포 파이프라인 계약 문제입니다.
2. `SERVER_URL`, context-path, 프론트 base path는 반드시 한 세트로 관리해야 합니다.
3. 301 리다이렉트는 POST를 깨뜨릴 수 있으므로 API 테스트 URL부터 HTTPS로 고정해야 합니다.

## ♻️ 재사용 체크리스트

- [ ] `VIRTUAL_HOST`, `VIRTUAL_PORT` 배포 스크립트 고정
- [ ] `SERVER_URL`에 context-path 포함 여부 점검
- [ ] `application-prod.yml` context-path와 프론트 base path 일치 확인
- [ ] 도메인 전환 시 CORS/쿠키 도메인까지 함께 점검

## ✨ 마무리

Nginx 문제를 빠르게 잡는 가장 좋은 방법은 "서버 문제"로 보지 않고 "요청 경로 시스템 문제"로 보는 것입니다.  
StoryG에서는 이 이슈를 계기로 배포 체크리스트를 URL 계약 중심으로 바꿨고, 비슷한 장애 재발이 크게 줄었습니다.
