# [StoryG 프로젝트] 시크릿 하드코딩 제거기

운영 이슈 문서를 정리하면서 가장 위험했던 패턴은 "일단 돌아가게 하려고 넣은 하드코딩 시크릿"이었습니다.  
JWT secret 기본값, DB root 계정 사용, Jenkinsfile 비밀번호 고정은 각각은 작은 편의였지만, 합쳐지면 큰 보안 리스크가 됩니다.

솔직히 말하면, 이건 저희 팀만의 문제가 아니었습니다. 많은 팀이 속도 때문에 같은 선택을 합니다.  
문제는 그 선택이 축적될 때입니다. 배포 편의의 합이 곧 공격 표면이 됩니다.

이번 작업의 핵심은 "문제 지적"이 아니라 "운영 가능한 대체 루틴"을 만드는 것이었습니다.  
그래서 시크릿 외부화, 최소 권한 계정, 파이프라인 주입 방식을 한 번에 묶어 재설계했습니다.

## 🕰️ 사건 타임라인

- 기능 이슈 점검 중 하드코딩 시크릿 다수 발견
- 위험도 분류(JWT/DB/Jenkins) 및 우선순위 결정
- 시크릿 외부화 + root 제거 + 파이프라인 정리 동시 진행
- 로테이션/롤백 절차 문서화로 운영 표준화

## 🚨 문제

StoryG에서 확인된 대표 보안 이슈:

- JWT secret 기본값 하드코딩
- DB root 계정 직접 사용
- Jenkinsfile에 DB 비밀번호 하드코딩

이 상태는 유출 시 즉시 악용 가능하고, 사고 시 영향 범위도 넓었습니다.

## 🧪 재현

보안 위험은 기능 장애처럼 오류 로그가 바로 나오지 않습니다. 대신 아래 조건에서 쉽게 재현됩니다.

1. 코드 저장소 접근 권한이 있는 계정이 시크릿 값을 조회 가능
2. 개발/운영 환경에서 동일 시크릿을 재사용
3. root 계정으로 애플리케이션이 광범위 권한 실행

즉, "문제 없어 보이는 상태"가 사실은 취약 상태였습니다.

## 🔍 원인 분석

근본 원인은 기술 부족보다 운영 습관이었습니다.

- 빠른 배포를 위해 시크릿을 코드에 임시 고정
- 환경별 시크릿 관리 체계 부재
- 권한 최소화 대신 root 계정으로 단순화

보안 이슈는 대개 코드 한 줄이 아니라, "관리 모델 부재"로 생깁니다.

## 🛠️ 해결

### 1) 시크릿 외부화

- JWT secret, DB password를 코드/리포지토리에서 제거
- 환경변수 또는 비밀 저장소를 단일 소스로 사용

예시:

```yaml
jwt:
  secret: ${JWT_SECRET}

spring:
  datasource:
    username: ${DB_APP_USER}
    password: ${DB_APP_PASSWORD}
```

### 2) root 계정 제거 + 최소 권한 계정 분리

- 앱 전용 DB 계정을 별도로 생성
- 필요한 스키마 권한만 부여

```sql
CREATE USER 'storyg_app'@'%' IDENTIFIED BY '***';
GRANT SELECT, INSERT, UPDATE, DELETE ON storyg_db.* TO 'storyg_app'@'%';
FLUSH PRIVILEGES;
```

### 3) Jenkins 파이프라인 정리

- Jenkinsfile 내 하드코딩 값 제거
- Credential Binding으로 주입
- 로그에 시크릿 노출되지 않도록 마스킹

### 4) 교체/롤백 전략 추가

- 시크릿 로테이션 절차 문서화
- 키 교체 시 서비스 영향 범위 사전 점검
- 장애 시 즉시 복구 가능한 롤백 경로 확보

## ✅ 검증

전환 후 아래를 검증했습니다.

- 저장소 검색 시 시크릿 문자열 미노출
- 앱이 root 없이 정상 동작
- Jenkins 로그에서 민감 정보 마스킹
- 키 교체 후 인증/DB 접속 정상

결과적으로 운영 보안 수준이 "개발 편의 중심"에서 "통제 가능한 운영 기준"으로 올라갔습니다.

## 📌 교훈

1. 시크릿은 코드가 아니라 운영 인프라에서 관리해야 한다.
2. root 계정은 편하지만, 사고 비용이 가장 비싼 선택이다.
3. 보안은 점검 항목이 아니라 배포 파이프라인의 기본 계약이다.
