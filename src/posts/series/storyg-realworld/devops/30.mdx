# [StoryG 프로젝트] `/api` vs `/api/v1` 전면 404 복구기

배포는 정상 완료됐는데, 프론트에서 호출하는 API가 전부 404를 반환하는 상황이 발생했습니다.  
문제의 핵심은 서버 장애가 아니라, **경로 계약(`/api` vs `/api/v1`)이 환경마다 달랐던 것**이었습니다.

현장에서 체감한 난이도는 꽤 높았습니다. 인프라 팀은 프록시를 의심했고, 백엔드 팀은 컨트롤러 매핑을 의심했고, 프론트는 캐시 문제를 의심했습니다.  
각자 부분적으로는 맞는 말이었지만, 전체를 보면 "계약 불일치"라는 하나의 문제였습니다.

이 글은 단순히 설정값을 무엇으로 바꿨는지가 아니라, 왜 경로를 "코드 설정"이 아니라 "조직 계약"으로 관리해야 하는지를 보여주는 기록입니다.

## 🕰️ 사건 타임라인

- 배포 직후 핵심 API 404 다수 관측
- 계층별 점검에서 증상 불일치(404/405 혼재) 확인
- 경로 계약 표준(`/api/v1`) 확정
- 프론트/백엔드/프록시/Swagger 동시 정렬 후 안정화

## 🚨 문제

- 프론트 요청: `/api/v1/...`
- 백엔드(prod) context-path: `/api`
- Swagger SERVER_URL 일부 설정도 `/api` 기준

결과적으로 라우팅 체인이 어긋나며 핵심 API가 전면 실패했습니다.

## 🧪 재현

다음 조합에서 동일 증상이 재현되었습니다.

1. 프론트 base API를 `/api/v1`로 고정
2. Spring prod context-path를 `/api`로 설정
3. 프록시/Swagger 설정이 혼합 상태로 배포

대표 증상:

```text
GET /api/v1/courses/public -> 404
POST /api/v1/auth/login   -> 404 또는 405
```

## 🔍 원인 분석

문제는 "한 파일 오타"가 아니라 다중 계층 계약 불일치였습니다.

- 프론트: 요청 경로 기준
- 백엔드: 실제 라우팅 기준
- Nginx: 프록시 전달 경로 기준
- Swagger: 테스트/문서 기준

이 네 계층 중 하나라도 다른 기준을 쓰면 배포 직후 전면 오류로 이어졌습니다.

## 🛠️ 해결

핵심은 경로를 고치는 게 아니라 **단일 계약을 정하고 전 계층을 일치**시키는 것이었습니다.

### 1) prod context-path 통일

```yaml
server:
  servlet:
    context-path: /api/v1
```

### 2) Swagger SERVER_URL 통일

```yaml
app:
  server-url: ${SERVER_URL:https://ourstoryg.kr/api/v1}
```

배포 환경변수도 동일하게 적용:

```bash
SERVER_URL=https://ourstoryg.kr/api/v1
```

### 3) 프록시/배포 체크리스트 고정

- `VIRTUAL_HOST`, `VIRTUAL_PORT`, `SERVER_URL` 일치 여부 점검
- 배포 후 smoke test를 `/api/v1` 기준으로 자동 실행

## ✅ 검증

복구 후 확인 항목:

- `GET /api/v1/courses/public` 정상 응답
- `POST /api/v1/auth/login` 정상 처리
- Swagger 테스트 경로와 실제 요청 경로 일치
- 프론트 라우팅/백엔드 라우팅 경로 계약 동일

결과적으로 "배포는 성공인데 API는 실패" 패턴이 크게 줄었습니다.

## 📌 교훈

1. API 경로는 코드 설정이 아니라 시스템 계약이다.
2. 경로 계약은 프론트/백엔드/프록시/Swagger를 한 번에 맞춰야 한다.
3. 배포 성공 로그보다 실제 경로 smoke test가 더 중요하다.

## ♻️ 재사용 체크리스트

- [ ] `context-path`, `SERVER_URL`, 프론트 base URL 일치
- [ ] Swagger 경로와 실제 엔드포인트 경로 일치
- [ ] 배포 후 GET/POST 핵심 API smoke test 실행
- [ ] 도메인/리다이렉트 변경 시 메서드 보존 점검
- [ ] 운영 런북에 "경로 계약" 항목 고정
